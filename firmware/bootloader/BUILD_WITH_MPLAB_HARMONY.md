# Build the bootloader and test application in MPLAB X/Harmony 3

## Why the project must be generated for the exact MCU

The Microchip `pic32mx_125_sk.X` example targets `PIC32MX570F512L`, which has much more Program Flash and SRAM than the PIC32MX534F064H. Copying its generated linker files unchanged can place data beyond the target's 16 KiB SRAM and can let the application exceed the target's 64 KiB Program Flash.

The correct process is to use the Microchip example as the functional reference, then have MCC generate startup, peripheral, USB, NVM, interrupt, and linker content for `PIC32MX534F064H`.

## Project A: USB HID bootloader

### A1. Create the project

1. Open MPLAB X IDE.
2. Select **File → New Project**.
3. Select **Microchip Embedded → Standalone Project**.
4. Device: `PIC32MX534F064H`.
5. Hardware Tool: PICkit 4, Snap, or Simulator for project creation.
6. Compiler: installed MPLAB XC32.
7. Project name: `bootloader_pic32mx534`.
8. Put it under `generated_projects`.

### A2. Open MCC

1. Open **Tools → Embedded → MPLAB Code Configurator**.
2. Install the Harmony 3 content requested by MCC.
3. Open the Microchip `pic32mx_125_sk.X` bootloader in a second MPLAB X window as a settings reference.

### A3. Add the required Harmony components

Add or confirm these components in the Project Graph:

- Core / Harmony Core
- NVM peripheral library
- USBFS driver
- USB Device Layer
- USB Device HID Function Driver, instance 0
- USB Device HID Bootloader
- GPIO and EVIC support generated by Core

The USB stack must be in **device** mode. Host support is not required.

Expected logical connections:

```text
USB Device HID Bootloader
        |
USB Device HID Function 0
        |
USB Device Layer
        |
USBFS Driver

USB HID Bootloader
        |
NVM PLIB
```

Use the auto-connect suggestions where MCC provides them.

### A4. Configure clocks for the 8 MHz crystal

Use MCC's clock configuration and verify the generated configuration bits.

Required clock relationship:

```text
Primary crystal = 8 MHz
System PLL input divider = 2
System PLL multiplier = 20
System PLL output divider = 1
SYSCLK = 80 MHz

USB PLL input divider = 2
USB PLL input = 4 MHz
USB clock = 48 MHz
```

Expected essential settings:

```c
FNOSC    = PRIPLL
POSCMOD  = HS
FPLLIDIV = DIV_2
FPLLMUL  = MUL_20
FPLLODIV = DIV_1
UPLLEN   = ON
UPLLIDIV = DIV_2
FPBDIV   = DIV_1
```

Copy `source_overrides/config_bits_8mhz.c` into the bootloader project. Do not keep a second file that defines the same configuration bits.

If the physical oscillator is not an 8 MHz crystal, stop and recalculate the PLL configuration.

### A5. Configure the USB descriptors

Use:

```text
Vendor ID:  0x04D8
Product ID: 0x003C
Device class/function: HID custom device
EP0 size: 64 bytes
```

The Unified Host setting must use Product ID `3C`.

### A6. Configure bootloader memory

Use a Program Flash bootloader with a 32 KiB reservation.

Required generated constants:

```c
FLASH_START       0x9D000000
FLASH_LENGTH      0x00010000
BOOTLOADER_SIZE   32768
APP_START_ADDRESS 0x9D008000
APP_JUMP_ADDRESS  0x9D008000
BTL_TRIGGER_LEN   16
```

For this PIC32MX family, the expected NVM geometry is:

```text
Write row:  128 words = 512 bytes
Erase page: 1024 words = 4096 bytes
```

The generated `bootloader_common.h` commonly represents those in words:

```c
PAGE_SIZE       128
ERASE_BLOCK_SIZE 1024
```

Required bootloader linker memory regions:

```ld
kseg0_program_mem : ORIGIN = 0x9D000000, LENGTH = 0x8000
kseg1_data_mem    : ORIGIN = 0xA0000010, LENGTH = 0x3FF0
```

MCC may emit `32768` instead of `0x8000`, and `16384 - 16` instead of `0x3FF0`; those are equivalent.

The first 16 bytes of SRAM are reserved for the boot request pattern.

### A7. Configure ICSP channel and development protections

Required during development:

```c
ICESEL = ICS_PGx1
JTAGEN = OFF
FWDTEN = OFF
PWP    = OFF
BWP    = OFF
CP     = OFF
```

`ICESEL = ICS_PGx1` matches PGEC1/PGED1 on MCU pins 15/16.

### A8. Generate and add the source override

1. Click **Generate** in MCC.
2. Replace generated `src/app.c` with `source_overrides/bootloader_app.c`.
3. Copy `source_overrides/bootloader_board.h` beside it.
4. Add `source_overrides/config_bits_8mhz.c` to the project.
5. Keep the MCC-generated Harmony bootloader, USB, USBFS, NVM, startup, and linker source.

The override implements:

- `0x5048434D` RAM request pattern
- optional force-boot switch
- optional bootloader-active LED
- default no-switch/no-LED operation when board pin aliases are not yet known

The generated weak `bootloader_Trigger()` is overridden by this source.

### A9. Build

Select **Production** build, then **Clean and Build Project**.

Inspect:

```text
dist/default/production/bootloader_pic32mx534.X.production.hex
dist/default/production/bootloader_pic32mx534.X.production.map
```

The bootloader's PFM sections must stay below `0x9D008000`.

Run:

```powershell
python scripts\verify_project.py --bootloader generated_projects\bootloader_pic32mx534.X
python scripts\hex_layout.py --mode bootloader path\to\bootloader.production.hex
```

## Project B: USB-loaded test application

### B1. Create the project

Create another standalone project:

```text
Device: PIC32MX534F064H
Project: test_app_pic32mx534
```

### B2. Configure its clock and application peripherals

The application runs after the bootloader and still needs the same 80 MHz clock runtime assumptions. However, the USB-loaded application HEX must not carry device configuration words.

Do not add `config_bits_8mhz.c` to this project.

### B3. Configure the bootloader-aware application linker

The correct target linker contract is:

```ld
PROVIDE(_ebase_address = 0x9D009000);

_RESET_ADDR = 0xBD008000;

kseg0_program_mem :
    ORIGIN = 0x9D009000,
    LENGTH = 0x7000

kseg1_boot_mem :
    ORIGIN = 0xBD008000,
    LENGTH = 0x490

kseg1_data_mem :
    ORIGIN = 0xA0000010,
    LENGTH = 0x3FF0
```

Meaning:

- First 32 KiB of PFM belongs to the bootloader.
- The application's reset entry uses the first page after the bootloader.
- EBASE and interrupt vectors begin on the next 4 KiB boundary.
- About 28 KiB remains for normal application code.
- First 16 SRAM bytes stay reserved.

Use the MCC bootloader application-linker generation feature where available. Compare its result to the contract above rather than pasting a linker from another MCU.

### B4. Add the visible test

1. Replace generated `src/app.c` with `test_app_snippets/test_app_app.c`.
2. Copy `test_app_snippets/test_app_board.h` beside it.
3. Map `TEST_APP_LED_*` macros to an actual LED pin.
4. Add `bootloader_request.c/.h` when the application must request a reset into bootloader mode.

### B5. Build but do not program with PICkit/Snap

Build the production HEX:

```text
dist/default/production/test_app_pic32mx534.X.production.hex
```

Run:

```powershell
python scripts\verify_project.py --application generated_projects\test_app_pic32mx534.X
python scripts\hex_layout.py --mode application path\to\test_app.production.hex
```

The application HEX must not contain:

- Program Flash below physical `0x1D008000`
- Boot Flash/config words
- Code beyond physical `0x1D00FFFF`

Load this HEX through Unified Host.

## Build-time acceptance checklist

- [ ] Device string is `PIC32MX534F064H`.
- [ ] No stale `PIC32MX570F512L` text remains in generated target files.
- [ ] CPU frequency is 80 MHz.
- [ ] USB clock is 48 MHz.
- [ ] Bootloader `FLASH_LENGTH` is `0x10000`.
- [ ] Bootloader reservation is `0x8000`.
- [ ] Application jump address is `0x9D008000`.
- [ ] Application EBASE is `0x9D009000`.
- [ ] RAM length is 16 KiB minus the 16-byte trigger.
- [ ] VID/PID is `04D8/003C`.
- [ ] Application HEX excludes configuration words.
- [ ] Both HEX layout checks pass.
